# ai-protal 执行流程 v2

## 目标

建立一套固定执行规范：

- 全自动闭环
- 仅关键节点汇报
- 故障连续自修 3 轮
- 每任务自动提交（若当前目录不是 Git 仓库，则标注未提交原因）

## 任务输入协议

用户最小输入只需一项：

1. `目标`

`验收标准`、`约束`、`非目标` 由执行方自动补齐，并在 `S0 接单` 节点先声明再执行。

默认自动补齐规则：

- 默认验收标准：
  - `npm run verify:gate` 通过
  - 目标路径功能可操作、无明显回归
- 默认约束：
  - 不改技术栈
  - 不引入新依赖（除非目标明确要求）
  - 不做破坏性环境操作
- 默认非目标：
  - 不做无关重构
  - 不做与目标无关的新功能扩展
  - 不修改未被目标触达的模块

如任务存在高风险或特殊边界，执行方可在 `S0` 按上下文覆盖上述默认值，并明确写出覆盖原因。

## 执行状态机

- `S0 接单`：重述目标，并自动生成 `验收标准/约束/非目标`
- `S1 探测`：非破坏性定位（读代码/日志/检查）
- `S2 实施`：最小改动实现
- `S3 验证`：执行固定验收闸门
- `S4 收口`：提交与结果汇报
- `SX 升级`：命中升级条件时暂停并向用户报告

## 关键节点汇报

仅保留三类节点汇报：

- `开始`
- `阻塞/升级`
- `完成`

完成汇报统一结构：

1. `结果`
2. `验证`
3. `提交`
4. `剩余风险`

## 故障自动修复策略

- 默认最多连续自修 `3` 轮
- 每轮固定顺序：`复现 -> 根因 -> 最小修复 -> 验证`
- 3 轮仍失败时进入 `SX 升级`，必须输出：
  - 已尝试项
  - 失败证据
  - 下一步建议

## 固定验收闸门

每次代码改动后必须执行：

1. `npm run build`
2. `npm test`
3. `aidc show -> aidc status -> aidc hide -> aidc status`
4. 若 GUI 条件可用，再执行 `npm run start` 冒烟

统一命令入口：

```bash
npm run verify:gate
```

若本机无 GUI 条件，必须在输出中明确标注跳过原因。
若需要强制 GUI 可用性（例如本机验收），使用：

```bash
AIDC_REQUIRE_GUI=1 npm run verify:gate
```

## 升级条件

命中以下任一条件必须暂停自动修复并向用户报告：

- 高风险改动：架构重排、依赖策略大改、TLS/代理/沙箱全局策略改动、数据/配置迁移
- 环境高风险：需要系统级权限、可能影响机器全局设置、潜在破坏性命令
- 需求冲突：当前任务与新目标无法在同一闭环中安全完成

## Git 规范

- 每任务自动提交
- 分支建议：`task/<yyyymmdd>-<slug>`
- 提交信息：`type(scope): summary`必须使用中文
- 提交正文固定包含：`Root cause`、`Changes`、`Verification`、`Impact`
- 自动提交脚本：`scripts/task-auto-commit.sh`

若当前目录不是 Git 仓库，则不做伪提交，只在完成汇报中说明未提交原因。

## 需求变更策略

用户执行中临时改需求时，立即中断当前流程并重排：

1. 先输出当前任务快照（已改动/未验证点/风险）
2. 再按新目标启动新闭环
